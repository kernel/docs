## How does CUA-powered login work?

Managed Auth uses CUA (Computer Use Agent) — an AI agent that visually interacts with web pages to perform login flows. When you start a login session, CUA:

1. Navigates to the website's login page
2. Discovers form fields, SSO buttons, and MFA prompts
3. Fills in credentials (from stored credentials or user input via the Hosted UI / Programmatic flow)
4. Handles multi-step flows, redirects, and CAPTCHAs
5. Confirms successful authentication

Because CUA adapts to each website's layout in real time, it works across sites without any site-specific configuration. You can watch the agent work via the `live_view_url` returned when starting a session.

## How does automatic re-authentication work?

When you link credentials to a connection, Kernel monitors the login session and re-authenticates automatically when it expires. Periodic health checks detect logged-out sessions and trigger CUA to re-auth in the background, so the profile stays logged in without additional action on your part.

<Warning>
Automatic re-authentication only works when the stored credentials are complete and don't require human input. If login needs SMS/email OTP, push notifications, or manual MFA selection, you'll need to trigger a new login session manually.
</Warning>


## How often are health checks performed?

Health checks run on regular cadences based on your plan. You can configure the interval with the `health_check_interval` parameter (in seconds) when creating a connection:

| Plan | Minimum interval | Default |
|------|-----------------|---------|
| Hobbyist | 3600s (1 hr) | 3600s (1 hr) |
| Startup | 1200s (20 min) | 3600s (1 hr) |
| Enterprise | 300s (5 min) | 3600s (1 hr) |

Maximum interval is 86400s (24 hours).

## How do I know if a Kernel can automatically re-authenticate a connection?

Check the `can_reauth` field on a connection. This boolean checks the following conditions:

1. **Credential linked** — A credential must be attached to the connection (stored in Kernel or via an external provider like [1Password](/integrations/1password))
2. **No external action required** — The learned login flow doesn't require human intervention

Only if all of the above conditions are met will `can_reauth` be `true`. When true, Kernel will attempt to automatically re-authenticate the connection.

### External actions that prevent auto-reauth

After a successful login, Kernel saves the login flow. If the flow includes steps that require human action—like SMS/email OTP, push notifications, or manual MFA selection—Kernel marks the connection as unable to auto-reauth because those steps can't be automated without user input.

If your login flow requires one of these, you can still automate around it:
- **Switch to TOTP** — If the site supports authenticator apps, add a `totp_secret` to your credential. TOTP codes are generated automatically, so the login flow won't require external action.
- **Trigger manual re-auth** — Start a new login session and route the user through the [Hosted UI](/profiles/managed-auth/hosted-ui) or [Programmatic](/profiles/managed-auth/programmatic) flow.

## Which authentication methods are supported?

CUA supports username/password authentication and most SSO providers. Common SSO provider domains (Google, Microsoft, Okta, Auth0, GitHub, Apple, Facebook, LinkedIn, OneLogin, Ping Identity, Amazon Cognito) are automatically allowed — you don't need to add them to `allowed_domains`.

<Warning>
Passkey-based authentication (e.g., Google accounts with passkeys enabled) is not currently supported. If a user's SSO provider requires a passkey, the login will fail.
</Warning>

## What happens if login fails?

If a login attempt fails, CUA will retry with exponential backoff. After multiple failures, the [login flow](/profiles/managed-auth/hosted-ui) will be marked as failed and you'll receive an error with a machine-readable `error_code`. Common failure reasons include:

- `credentials_invalid` — Invalid credentials
- `bot_detected` — Bot detection blocking the login page
- `captcha_blocked` — CAPTCHAs that couldn't be solved
- `login_form_not_found` — CUA couldn't find a login form on the page

CUA also detects errors displayed by the website itself (e.g., "Incorrect password") and surfaces them in the `website_error` field. See the [error codes reference](/profiles/managed-auth/programmatic#error-codes) for the full list.

## Can I use Managed Auth with any website?

CUA works with most websites by adapting to each site's login flow. Sites with aggressive bot detection may require additional configuration (stealth mode, proxies). Passkeys and hardware security keys are not currently supported.

## Can I watch the login happening?

Yes. When you start a login session, the response includes a `live_view_url` — a browser live view URL that lets you watch the CUA agent interact with the login page in real time. This is useful for debugging failed logins or understanding how a particular site's flow is handled.

The `live_view_url` is also available on the connection object while a flow is in progress.

## How is Managed Auth billed?

Managed Auth is included on all paid plans with no per-connection fees. It uses browser sessions to log in and keep your sessions fresh—these count toward your browser usage like any other browser session.

Auth sessions are fast (typically 5-30 seconds each). Kernel monitors session health and re-authenticates automatically when sessions expire—most stay valid for days. For example, keeping 100 auth connections logged in typically costs less than $5/month in browser usage. See [Pricing & Limits](/info/pricing#managed-auth) for details.
